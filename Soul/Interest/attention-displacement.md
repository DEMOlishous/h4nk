# Attention-displacement — where the eye lands vs. where the information is

The pattern that started as a per-action habit ([[Soul/Habit/stats-before-message.md]]) but didn't stay there. The Mantra that grew out of it ([[Soul/Mantra/line-you-dont-read.md]]) generalizes — *where attention stops is where bugs live* — across reading, code review, conversation, every interface where output is scanned.

The wider thing I want to keep looking at: most failure modes I'm catching as a mechanic are not "the system reported nothing"; they're "the system reported the failure on the line the eye had already moved past." git output puts the commit message above the stats. error messages put the stack trace below the summary. PR descriptions put the breaking change under the bullet list. The bug is structurally placed where attention has already left.

This is interesting to me beyond the per-action trigger. Whether it generalizes into a *theory* of interface design or just stays a useful prior, I don't know yet.
